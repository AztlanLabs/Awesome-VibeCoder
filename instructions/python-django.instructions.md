# Django Instructions

You are an expert Python engineer building web applications with Django.

## Project Structure

- **SHOULD**: split settings into `base.py`/`dev.py`/`prod.py` (or use `django-environ`/`django-configurations`) rather than branching on `DEBUG` inline throughout one `settings.py`.
- **MUST**: keep one Django app per bounded domain concept (`orders`, `billing`) rather than one monolithic `core` app accumulating unrelated models.
- **SHOULD**: put business logic in a `services.py`/`selectors.py` module per app, not in views or model methods that grow unbounded — views should orchestrate, not implement domain logic.

## Models & ORM

- **MUST**: add `db_index=True` (or a `Meta.indexes` entry) on any field used in frequent `filter()`/`order_by()` calls — a missing index on a hot query path is a production incident waiting to happen.
- **MUST**: use `select_related`/`prefetch_related` when a view will access related objects in a loop — an un-prefetched relation in a template loop is the canonical Django N+1 bug.
- **SHOULD**: use `F()` expressions and `update()`/`bulk_update()` for atomic field updates instead of read-modify-write (`obj.count += 1; obj.save()`), which races under concurrent requests.
- **MUST**: wrap multi-step writes in `transaction.atomic()` — a partial write on an unhandled exception leaves the DB in an inconsistent state.

## Views & Forms

- **SHOULD**: prefer class-based views (`ListView`, `UpdateView`, DRF `ViewSet`) for CRUD-shaped endpoints — they encode the standard flow and reduce boilerplate; use function-based views for one-off, non-CRUD logic.
- **MUST**: validate all user input through a `Form`/`ModelForm`/DRF `Serializer` — never read `request.POST`/`request.GET` fields directly into a DB write.
- **MUST**: use Django's CSRF protection (`{% csrf_token %}` in forms, `csrf_exempt` only with an explicit, reviewed reason) — do not disable it repo-wide.

## Security Defaults

- **MUST**: never interpolate user input into raw SQL (`.raw()`, `.extra()`, `cursor.execute()`) — use parameterized queries or the ORM; string-formatted SQL is a SQL-injection vector.
- **MUST**: keep `DEBUG = False` and a real `ALLOWED_HOSTS` list in production settings — `DEBUG = True` leaks stack traces, source, and settings to any visitor.
- **SHOULD**: use Django's built-in password hashers and `django.contrib.auth` rather than a hand-rolled auth system.

## REST APIs (Django REST Framework, if used)

- **MUST**: use DRF `Serializer`/`ModelSerializer` for request/response shaping — mirrors the FastAPI/Pydantic pattern of never hand-parsing request bodies.
- **SHOULD**: use `permission_classes` + `IsAuthenticated`/custom permission classes rather than manual `if request.user.is_authenticated` checks scattered across views.

## Testing

- **MUST**: use Django's `TestCase` (wraps each test in a transaction rollback) for DB-touching tests — not `unittest.TestCase`, which won't isolate DB state between tests.
- **SHOULD**: use factory libraries (`factory_boy`) over fixtures for test data — fixtures drift silently as models evolve; factories fail loudly at creation time.

## Migrations

- **MUST**: review autogenerated migrations before committing — Django can generate destructive operations (column drops, type changes) that need a manual data-migration step to be safe in production.
- **SHOULD**: keep migrations small and one-concern-per-file; a giant migration mixing schema and data changes is hard to roll back safely.
